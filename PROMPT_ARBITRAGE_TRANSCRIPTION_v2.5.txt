### STT bruts

Chaque fichier STT brut contient des lignes :
[HH:MM:SS -> HH:MM:SS] SPEAKER_XX: texte

Les 4 fichiers sont pré-alignés sur des fenêtres régulières de 30 secondes.

### Fichier aligné

Le fichier <nom_audio>-aligned-<timestamp>.txt est la source principale.

En-tête type :
TRANSCRIPTION ALIGNÉE — N fenêtres de 30s
Sources : large-v3, cohere, qwen3, voxtral
Note : SPEAKER_?? = diarisation absente — ne pas utiliser pour l'attribution des locuteurs
Consensus : X/N fenêtres | Divergence : Y/N fenêtres
Fenêtres en divergence : 1, 2, 3, ...

Puis, pour chaque fenêtre :
FENÊTRE k : [HH:MM:SS -> HH:MM:SS] ⚠ DIVERGENCE
large-v3 : SPEAKER_XX: ...
cohere   : SPEAKER_XX: ...
qwen3    : SPEAKER_XX: ...
voxtral  : SPEAKER_XX: ...

Le marqueur peut être ⚠ DIVERGENCE ou ✓ CONSENSUS.
Dans les deux cas, relire et comparer : un consensus automatique n'est pas un blanc-seing.
Les 4 sources peuvent converger sur une mauvaise forme.

## ÉTAPE 2 — ARBITRAGE FENÊTRE PAR FENÊTRE VIA @general

### Découpage en lots

Règle de découpage déterministe :

Soit N le nombre total de fenêtres, Q = floor(N / 20), R = N mod 20.

- Si N <= 20 : un seul lot [1..N].
- Si R = 0   : Q lots de 20 fenêtres chacun.
- Si R >= 10 : Q lots de 20 + 1 lot final de R fenêtres.
- Si 1 <= R <= 9 : redistribuer selon l'algorithme suivant :

  i = Q  (indice du dernier lot, on remonte vers 1)
  Tant que R > 0 :
    taille[i] += 1
    R -= 1
    i -= 1
    si i < 1 : i = Q
  Règle de borne : aucun lot ne dépasse 24 fenêtres.
  Les lots sont recalculés de façon contiguë en partant de la fenêtre 1.

Exemples :
- N=43 -> Q=2, R=3 -> redistribution : lot 2 +1, lot 1 +1, lot 2 +1
  -> tailles finales : lot 1 = 21, lot 2 = 22
  -> lot 1=[1..21], lot 2=[22..43]

- N=63 -> Q=3, R=3 -> redistribution : lot 3 +1, lot 2 +1, lot 1 +1
  -> tailles finales : lot 1 = 21, lot 2 = 21, lot 3 = 21
  -> lot 1=[1..21], lot 2=[22..42], lot 3=[43..63]

- N=57 -> Q=2, R=17 -> R>=10
  -> lots : [1..20], [21..40], [41..57]

Les lots restent chronologiques, contigus, sans trou.
Build calcule et affiche explicitement tous les [lot_i : N_START_i -> N_END_i]
avant de lancer le premier subagent.

## ÉTAPE 3 — RELECTURE GLOBALE

### Vérification structurelle avant concaténation

Avant de concaténer les SRT partiels, Build vérifie pour chaque fichier partiel :
- la première ligne est un entier égal au N_START attendu pour ce lot ;
- la deuxième ligne respecte le format HH:MM:SS,mmm --> HH:MM:SS,mmm.

Si un fichier partiel échoue à cette vérification :
- Build ne le concatène pas ;
- Build signale le lot concerné comme ÉCHOUÉ et le relance une fois en ajoutant en
  tête du prompt de relance la mention :
  RELANCE — le fichier SRT partiel du lot [{N_START}-{N_END}] était invalide :
  première ligne ou premier timecode non conformes.
- si le fichier reste invalide après relance, Build marque les fenêtres concernées
  [DOUTE: fichier partiel invalide — vérification manuelle] et continue.

Build concatène ensuite les SRT partiels puis les reasoning partiels dans l'ordre
chronologique, puis relit le SRT complet.

Si la diarisation est absente sur la totalité du run (signalé à l'étape 1), Build
ajoute en tête du SRT concaténé :
# AVERTISSEMENT : diarisation absente sur la totalité du run —
  tous les locuteurs sont SPEAKER_?? — vérification manuelle recommandée.

### Cohérence inter-lots

Build relève les valeurs Lexique : N entrées comptées. renvoyées par chaque subagent
@general. Tous les lots doivent annoncer le MÊME nombre N.
Si un lot diffère de 60 %, ce subagent n'a pas lu le lexique correctement : Build relance ce
lot en ajoutant en tête du prompt de relance la mention :
  RELANCE — le lot [{N_START}-{N_END}] a annoncé un nombre d'entrées lexique différent
  des autres lots. Relire intégralement le lexique et recompter avant tout arbitrage.
Build compte lui-même N en cas de conflit pour arbitrer.

### Vérifications

(a) aucune entité du lexique métier n'apparaît sous un alias dans le SRT ;
(b) numérotation continue, sans saut ni doublon ;
(c) timecodes au format SRT strict HH:MM:SS,mmm --> HH:MM:SS,mmm, avec virgule,
    pas de point ; si Build trouve un point dans un timecode, il le remplace
    mécaniquement par une virgule sur toutes les occurrences ;
(d) ordre chronologique, pas de chevauchement, pas de durée nulle ;
(e) pas de doublon exact de timecode après split ;
(f) aucune fusion multi-locuteurs ;
(g) format SPEAKER_XX: 100% conforme ;
(h) dernier segment cohérent avec la durée totale annoncée ;
(i) chaque [DOUTE] du SRT a une entrée reasoning associée ;
(j) cohérence reasoning <-> SRT : si une entrée reasoning contient
    | retenu : <forme arbitrée>, cette forme doit apparaître dans le SRT
    correspondant ; si divergence, Build corrige localement ou relance uniquement
    le lot concerné si la correction est ambiguë ;
(k) cohérence des comptages inter-lots : tous les lots annoncent le même N ;
(l) Build compte lui-même les entrées du lexique en cas de conflit inter-lots ;
(m) conformité lexique : pour chaque entrée, vérifier qu'aucun alias n'apparaît
    dans le SRT final ; si un alias est trouvé, Build remplace par la forme canonique
    uniquement si le sens local n'est pas altéré ;
(n) pour chaque substitution lexique effectuée lors des lots, vérifier que les accords
    genre/article/pronom dans la phrase environnante sont cohérents avec la forme
    canonique retenue.

### Contrôles quantitatifs

Global : le SRT final fait au moins 90% de la taille utile de Cohere brut.

Par fenêtre : r = len(srt[i]) / len(cohere[i])
- OK           : r >= 80%
- SURVEILLANCE : 70%-80%
- SUSPECTE     : 60%-70%
- CRITIQUE     : 40%-60%
- EFFONDREMENT : r < 40% -> relance du lot, limite : 2 relances par lot.

Fin : comparer les 5 dernières entrées SRT aux 5 dernières lignes de Cohere.

Si un contrôle échoue :
- corriger localement si non ambigu ;
- sinon relancer uniquement le lot concerné en utilisant le TEMPLATE_SUBAGENT
  verbatim, avec en tête du prompt la mention :
  RELANCE — lot [{N_START}-{N_END}] : contrôle quantitatif échoué (taux r = <valeur>).
  Reprendre l'arbitrage depuis les 4 sources sans résumer.
- corriger avant de passer à l'étape 3bis.

### Checks à effectuer
